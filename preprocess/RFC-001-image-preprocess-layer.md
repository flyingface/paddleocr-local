# RFC-001: 引入图像前置层(Preprocess Layer)

> **作者**: flyingface
> **日期**: 2026-06-27
> **状态**: Draft v0.3
> **目标读者**: 团队评审 + 后续接手人

---

## 摘要

在 `paddleocr-local` 的 PaddleOCR 推理前,引入一个**图像前置层** (`preprocess/`),
把"OCR 之前的不确定性"全部消化掉:几何矫正 + 业务分类 + 智能路由。
**Pandocr-web 的核心业务代码改动 < 30 行。**

## 动机

### 现状

`paddleocr-local` 假设输入是"干净印刷件"。
现实业务中(卡证、票据、合同、眼镜第一视角拍照) **30-50% 的输入不满足假设**,
导致:

- 字段抽取准确率掉 10-20%
- 边界场景翻车概率高
- 团队每次新场景都要改 OCR 参数

### 目标

建立一个**可插拔、可观测、可演进**的图像前置平台,
让下游 OCR 只做最擅长的事。

## 设计

### 架构

```
file → pandocr-web → preprocess-gateway → 子服务群 → 路由决策 → OCR
                                ├─ correction (resnet18)
                                ├─ classify (PaddleClas)
                                └─ quality (可选)
```

详细见 `preprocess/ARCHITECTURE.md`。

### 关键决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 颗粒度 | 微服务而非单体 | 模型独立扩缩,故障隔离 |
| 编排方式 | Gateway + YAML | 规则可视化、可热改 |
| 接入策略 | 渐进式,通过环境变量开关 | 不破坏现有功能 |
| 失败行为 | 降级到无预处理 | 永远不能让 OCR 流程崩 |
| GPU 资源 | 矫正用 GPU,分类用 GPU | 模型选择决定 |
| 数据存储 | 无状态,只走 ConfigMap | 简单 |

## 风险

| 风险 | 等级 | 缓解 |
|---|---|---|
| 矫正模型 ONNX 导出失败 | 🟧 | 用 PaddleX 原生 + 备用 resnet18 PyTorch |
| Pandocr-web 接入引入 bug | 🟧 | 渐进式 + 强降级 |
| 分类服务训练数据不足 | 🟧 | LLM 合成 + 公开集 + 业务真图 |
| 910B NPU 不支持 | 🟥 | 跳过,4090 + K8s 走 |
| 跨机房网络延迟 | 🟧 | 预处理和 OCR 在同集群 |

## 替代方案

### 方案 A:把所有模型塞进 pandocr-web

- ❌ 镜像爆炸
- ❌ 升级耦合
- ❌ 故障域大

**不采用。**

### 方案 B:用 LLM 多模态直接做

- ❌ 延迟高(1-3s)
- ❌ 成本高
- ❌ 不可控

**不采用,作为长期演进方向(v2.0 智能路由)。**

### 方案 C:接外部 SaaS OCR

- ❌ 数据出域
- ❌ 不可控
- ❌ 长期成本高

**不采用。**
